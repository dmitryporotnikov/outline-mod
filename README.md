# outline-mod

A modified [Outline VPN](https://getoutline.org) server that routes all client traffic through a WireGuard tunnel. Two hops instead of one.

## Why

Some countries apply internet censorship unevenly. Connecting to a domestic IP often works fine. The filters either skip local traffic or don't look at it as hard. But connecting directly to a server in Germany or the US gets flagged and blocked.

So: rent a cheap VPS inside your country (or a neighboring one with lax filtering toward your IP range). Your Outline clients connect to that VPS, which looks domestic to the filters. The VPS forwards everything through a WireGuard tunnel to a server abroad, where the traffic actually exits to the open internet.

```
You  -->  Domestic VPS (Outline + WireGuard)  -->  Foreign VPN server  -->  Internet
        looks like local traffic                  actual exit point
```

The censorship system sees a connection to a local IP on port 443. Looks like ordinary HTTPS. What happens after that is none of its business.

## How it works

The setup runs two containers sharing a network namespace:

1. **wireguard-proxy** connects to your WireGuard VPN provider and handles all the routing. It uses CONNMARK iptables rules to make sure responses to incoming connections go back through Docker's network (not through the tunnel), while outgoing proxy traffic goes through WireGuard.

2. **shadowbox** (Outline's server component) handles Shadowsocks encryption and client management. Because it shares the wireguard container's network stack, all its outgoing traffic automatically goes through the tunnel.

Port 443 is used for client traffic (looks like HTTPS), and port 8090 for the management API.

## Prerequisites

You need a VPS with Docker and Docker Compose installed, and a WireGuard config file from a VPN provider. The VPS should have ports 443 and 8090 accessible from outside.

## Setup

1. Clone this repo and copy your WireGuard config:

```bash
git clone https://github.com/youruser/outline-mod.git
cd outline-mod
cp /path/to/your/wireguard.conf vpn.conf
```

2. Patch the WireGuard config to add the required routing rules:

```bash
./patch_vpn_conf.sh vpn.conf
```

This adds iptables PostUp/PostDown rules that the container needs for proper traffic routing. If your config already has PostUp rules, the script skips them.

3. Run the setup:

```bash
sudo ./setup.sh --wireguard-config /path/to/vpn.conf
```

4. The script prints a JSON blob at the end. Paste it into the Outline Manager app (step 2 of "add server").

## Options

```
--wireguard-config PATH    Path to your WireGuard config file (required)
--api-port PORT            Management API port, default 8090
--hostname HOSTNAME        Public IP or hostname, auto-detected if not set
```

## Files

```
setup.sh              Main setup script
cleanup.sh            Stops containers and removes state (--full to wipe everything)
patch_vpn_conf.sh     Adds required iptables rules to a WireGuard config
docker-compose.yml    Container definitions (also generated by setup.sh)
vpn.conf              Your WireGuard config (you provide this)
```

## Troubleshooting

Check if the WireGuard tunnel is actually passing traffic:

```bash
docker exec wireguard-proxy wg show
docker exec wireguard-proxy curl -s --max-time 5 ifconfig.me
```

`wg show` should have a "latest handshake" line and nonzero received bytes. The curl should return your VPN provider's IP, not your server's IP.

If the tunnel shows 0 bytes received, your VPN provider's endpoint is unreachable from the VPS. Try a different server or port from your provider.

Check Shadowbox logs:

```bash
docker logs shadowbox
docker logs wireguard-proxy
```

To start over:

```bash
sudo ./cleanup.sh --full
sudo ./setup.sh --wireguard-config /path/to/vpn.conf
```

## How the routing works

This took a while to get right. WireGuard's `AllowedIPs = 0.0.0.0/0` sends all traffic through the tunnel, including responses to incoming client connections. If a client connects to your server and the response goes out through WireGuard, it arrives from the wrong IP and gets dropped. Useless.

The PostUp rules in vpn.conf fix this with CONNMARK. At startup, the rules detect whichever interface carries the default route (usually eth0, but not always) and tag incoming connections on it. When the container responds on a tagged connection, a policy routing rule forces the response back through that interface. New outgoing connections (the proxied traffic) don't have the tag, so they go through the tunnel. It's a bit ugly but it works.
